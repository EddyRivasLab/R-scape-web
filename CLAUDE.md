# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A web front-end for **R-scape** (RNA Structural Covariation Above Phylogenetic
Expectation), a command-line bioinformatics tool. The web app lets users upload an
RNA multiple sequence alignment (Stockholm format), runs the `R-scape` binary on the
server, and renders the resulting covariation tables and SVG plots.

It is a **Perl Catalyst** application (Catalyst::Runtime 5.90+, Moose-based). The
R-scape binary itself is C, shipped as `rscape.tar.gz` and compiled inside the
container at build time — it is *not* part of this repo's source.

## Repository layout

- `R-scape/` — the actual Catalyst application (this is the app root; most work happens here).
- `Dockerfile` / `docker-compose.yml` — build and run the app in a container.
- `rscape.conf.dev` — config mounted over the app's `rscape.conf` during local dev.
- `rscape.tar.gz` — the R-scape C source tarball, compiled at image build time.
- `assets/` — logo source files (not served).

Files ending in `~` are stray editor backups; ignore them and never edit them.

## Build and run

The app is designed to run only inside the container (it shells out to the compiled
`R-scape` binary, `gnuplot`, etc.). Use **podman**, not docker.

```bash
# Build the image (compiles R-scape from rscape.tar.gz; slow).
podman build -t eddylab.org/rscape:2.6.8 .

# Run via compose. Mounts ./R-scape into /app and rscape.conf.dev over the config,
# so code/template edits are live without rebuilding. Exposes port 8080.
podman-compose up
```

The dev server is launched by the Dockerfile `CMD` as
`perl ./script/rscape_server.pl -d -r -p 8080` (`-d` debug, `-r` restart on file
change). **Do not start dev servers yourself** — the user runs them.

The bundled startup scripts in `R-scape/script/` (`rscape_server.pl`,
`rscape_fastcgi.pl`, `rscape_cgi.pl`, `rscape_create.pl`) are standard Catalyst
scaffolding generated by the framework — rarely edited.

## Tests

Standard Catalyst/Perl test suite in `R-scape/t/`. Run inside the container (deps and
the Catalyst app must load):

```bash
# All tests
perl Makefile.PL && make test

# A single test file
perl -Ilib t/controller_Results.t
```

`t/01app.t` loads the app; `t/02pod.t` / `t/03podcoverage.t` are POD checks; the
`controller_*` / `model_*` / `view_*` files are per-component stubs.

## Architecture & request flow

Standard Catalyst MVC. Components live under `R-scape/lib/Rscape/`.

**The core logic is in two places — read these first:**

1. `lib/Rscape/Model/Rscape.pm` — `run()` is the heart of the app. It:
   - Saves the upload to a fresh `tempdir` under `dir_path` (configurable;
     `/tmp/rscape` in dev) as `uploaded.sto` (renamed deliberately so users can't
     control the path).
   - Detects whether the alignment has a consensus structure (`#=GC SS_cons`).
   - Serializes request metadata (`evalue`, `mode`, `has_ss_cons`, `upload_name`) to
     a `meta` file via Sereal, alongside a `cmd` file recording the exact command run.
   - Builds and `system()`-executes the `R-scape` binary, then returns the tempdir's
     basename, which becomes the result ID in URLs.

2. `lib/Rscape/Controller/Results.pm` — handles submission and result rendering:
   - `POST /results` (`index` → `process`) runs the model and redirects to
     `/results/<id>`.
   - `GET /results/<id>` (`results`) reads output files back from the tempdir and
     stashes them for the template.
   - `GET /results/<id>/<type>/...svg` (`svg_images`) streams the right SVG
     (`his`/`dplot`/`r2r`) by globbing the output dir. R-scape draws its gnuplot
     figures in layers, and gnuplot's SVG terminal writes each layer as a
     *separate* stacked SVG document in one file (e.g. 6 for `surv`/his, 3 for
     `dplot`). Browsers only render the first, so `svg_images` merges the layers
     back under a single `<svg>` root before serving. The R2R plot is a single
     document (from R2R, not gnuplot) and passes through unchanged.

`lib/Rscape/Controller/Download.pm` serves the `.cov` and `.power` files as plaintext
attachments. `lib/Rscape/Controller/Root.pm` is the `/` landing page and 404 handler.

### The "mode" concept (pervasive — get this right)

A request's `mode` (1–4) drives the R-scape command flags *and* which output files to
read *and* how templates render. The mapping appears in `Model/Rscape.pm::run`:

- `1` — One-set test (`R-scape`)
- `2` — One-set test + report a structure (`R-scape --cacofold`)
- `3` — Two-set test (`R-scape -s`) — **requires** `SS_cons` in the upload
- `4` — Two-set test + report a structure (`R-scape -s --cacofold`) — **requires** `SS_cons`

For modes 3/4, the model returns the string `'alignment_missing'` early if the upload
lacks `SS_cons`, which the controller turns into a redirect to the error page.

**Modes 2 and 4 read `.cacofold.*` output files** (e.g. `output.cacofold.cov`,
`output.cacofold.power`) instead of the plain `.cov`/`.power`, and render extra table
columns. This `$mode =~ /^(2|4)$/` branch is duplicated across `Results.pm`,
`Download.pm`, and `results/results.tt` — change all of them together.

### Output file naming

The model runs R-scape with `--outname output`, so result files are always
`output.cov`, `output.power`, `output.cacofold.cov`, etc.
`determine_name_from_sto_file()` returns the constant `"output"` and short-circuits
before its legacy ACC/ID-parsing code (kept for reference only). Note: `Download.pm`'s
`read_results`/`read_power_results` strip a file extension off the name and contain a
fallback to older `.fold.*` filenames — a quirk to preserve when touching downloads.

### Views

Template Toolkit (`.tt`) views under `R-scape/root/`, served by
`lib/Rscape/View/HTML.pm`. All pages render inside `root/layouts/wrapper.tt`
(Bootstrap 3 + jQuery from CDNs). The home page form is `root/index.tt`; results are
`root/results/results.tt`. Error pages (`bad_input.tt`, `no_results.tt`,
`not_found.tt`, `alignment_missing.tt`) are selected by `:Private` actions that just
set `$c->stash->{template}`. Static assets (Bootstrap, jQuery tablesorter, `app.js`,
`app.css`) live in `root/static/` and are served by `Static::Simple`.

## Configuration

Catalyst loads `rscape.conf` (Config::General format) from the app home dir via
`ConfigLoader`. The key block is `<Model Rscape>`, which sets:

- `dir_path` — where uploads/results tempdirs are created (`/tmp/rscape`).
- `rscape_dir` — install path of the compiled R-scape (`/build/rscape_v2.6.8` in the
  container; this is also exported as `RSCAPE_HOME`).
- `gnuplot` / `gnuplot_ps` — gnuplot binary and PostScript dir, exported as env vars
  before running R-scape.

There are several variants: `rscape.conf.dev` (mounted in dev), `rscape.conf.live`
(production paths), `rscape.conf.test`. The `using_frontend_proxy` /
`ProxyBase` request trait in `lib/Rscape.pm` means the app expects to sit behind a
reverse proxy (nginx) in production and rewrites generated URLs accordingly.

## Things to watch out for

- **Version bumps**: the R-scape version (`2.6.8`) is hard-coded in three spots that
  must stay in sync — the image tag in `docker-compose.yml`, and the
  `rscape_v2.6.8` directory name in both the `Dockerfile` (`WORKDIR`) and
  `rscape.conf.dev` (`rscape_dir`). The C source dir name comes from `rscape.tar.gz`.
- The model builds a shell command string and runs it with `system()`. Uploads are
  saved under a server-controlled filename specifically to avoid path injection — keep
  that protection when modifying `run()`.
- This is an old codebase (Bootstrap 3, jQuery 1.12, Catalyst). Match the surrounding
  style; don't introduce modern build tooling.
- **The image base is `ubuntu:18.04`** (the Catalyst stack is installed from apt;
  only `CatalystX::RoleApplicator` and `Catalyst::TraitFor::Request::ProxyBase` come
  from CPAN). gnuplot must be **5.2+** — 5.0's SVG terminal emitted invalid XML
  (unbalanced `<g>`), so don't downgrade the base. On Apple Silicon the build runs
  natively for arm64; the bundled R-scape `config.sub`/`config.guess` are too old to
  recognize aarch64, so the Dockerfile refreshes them from `autotools-dev`.
- The `rscape.tar.gz` is packaged from a macOS tree and ships stale Mach-O `.o`/`.a`
  files; the Dockerfile deletes them before `make` so they're rebuilt for Linux.
